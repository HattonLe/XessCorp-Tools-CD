Problem with clearing D flip-flop

Q:

Hi everyone, I’m trying to get the D flip-flop with asynchronous clear from the Xilinx Student Edition to work. I’ve made a few small mods to the code as shown below. The data appears to be clocking out OK but for some reason the active low async clear isn’t working. I’ve compared the code to Xilinx’s synthesis templates and checked the XESS Faq list/ xilinx.com site and cannot see why there’s a problem. Am I missing something obvious here??

-- edge-triggered D flip-flop with asynchronous clear
LIBRARY IEEE;
USE IEEE.std_logic_1164.all;

-- edge-triggered flip-flop interface description
ENTITY msdac IS
PORT
(
c: IN STD_LOGIC;        -- clock
d: IN STD_LOGIC;        -- data input
aclrb: IN STD_LOGIC;    -- active-low asynchronous clear input
sq: BUFFER STD_LOGIC    -- data output
);
END msdac;

-- edge-triggered flip-flop architecture description
ARCHITECTURE msdac_arch OF msdac IS
COMPONENT IBUF PORT (I: IN STD_LOGIC; O: OUT STD_LOGIC); END COMPONENT;
COMPONENT BUFG PORT (I: IN STD_LOGIC; O: OUT STD_LOGIC); END COMPONENT;
SIGNAL c_in, c_buf: STD_LOGIC;  -- buffered, low-skew clock signals
BEGIN

u0: IBUF PORT MAP (I=>c, O=>c_in);
u1: BUFG PORT MAP (I=>c_in, O=>c_buf);

PROCESS (aclrb,c_buf)
BEGIN
-- clear the flip-flop if the asynchronous clear input is low
IF aclrb='0' THEN
  sq <= '0';
-- otherwise store the value on the d input at the falling clock edge
ELSIF (c_buf'event AND c_buf='0') THEN
  sq  <= d;
-- you don't need the following two lines since it is implied that
-- sq will not change if none of the previous statements is active,
-- but it doesn't hurt to add them
--ELSE
--  sq <= sq;
END IF;
END PROCESS;

END msdac_arch;

A:

I entered your VHDL code into Foundation 2.1i with SP6. It synthesized OK and I checked the synthesized circuit and it was OK as well. However, the functional simulation did not work. The asynchronous clear would not set the flip-flop output to zero. Then I implemented it and downloaded it to an XS40 Board and it worked correctly. So it appears to be a problem with the simulator. The XNF file seems to use the startup block to do a global reset of the flip-flops and (possibly) this is not being simulated correctly. I don't have a solution for this problem at this time.
