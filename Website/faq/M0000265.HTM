Getting RANDGEN to work reliably

Q:

OK, I think I just figured out something useful. I could NOT get the randgen (or any other 8031) application to work reliably, the 8031 or 4010 would just not start up reliably.

What I found was that after I unplugged the power from the xs40, the led's were still lit. After running gxsport, I saw that xsload had left some parallel port pins high. Somehow this messes things up, because by some path I don't understand, the xs40 still gets powered through the parallel port.

What I discovered was that if you write a script to reset then run the 8031 immediately after loading the hex and bit files, then randgen reliably starts up and happily runs. I put the folllowing in .cmd file:

xsload rnddisp.hex tdrandg.bit
xsport 1
xsport 0

I think this is a bug in xsload, it seems to me that it should leave the parallel port in a known state, preferably high impedance.

A:

If you look in XSTEST.BAT, you will see the batch file ends with the same XSPORT sequence you described. You always need to do an explicit reset for the 8031 after the FPGA is programmed because you cannot be sure what happens to the 8031 during the FPGA configuration sequence.
