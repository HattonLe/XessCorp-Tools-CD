Single-stepping state machines

Q:

I need some advice. I'm implementing a rather involved statemachine in VHDL on an XS40-010E. My code consists of a process with a big case statement and a process that advances the state when the clock ticks. I needed to single cycle it for debugging, so I modified the clocking process to require data from the pc parallel port. Since I could control this from the keyboard, it seemed to be just what I needed.

What I got was very strange behavior. Things would happen almost randomly. Signals would latch in the wrong state, and the state machine would transition to wrong states.

Eventually the light dawned when I tried to remove the clock from the sensitivity list of the clocking process. At that point the compiler complained that the thing I was using for a clock had to come in on a clock pin. I think that what was happening was that the "clock" signal I generated was not being distributed to all parts of the chip simultaneously, fooling parts of the machinery.

Now that I've explained all that, I have two questions: 

  1. Is this a plausible assessment? and 
  2. Is there a reliable way to single cycle the state machine from VHDL (ie without physically modifying the connection to the clock chip)?

A:

Normally, you should use one combinatorial process with the case statement to generate the next state from the current state and any other inputs to the circuit. Then you should have a very simple second process which transfers the next state into the current state on the rising or falling clock edge.

In order to single-step, you should drive the second process with a clock from the first or second bit of the parallel port data pins. The signals on these pins pass through Schmitt-trigger inverters which clean the ragged edges of the transitions after passing through the parallel port cable. Then the parallel port clock signal passes through an IBUF and a BUFG. The output of the BUFG becomes the clock for the second process.

When you are done debugging, change the pin assignment for the clock from the parallel port pin to whatever pin the real clock is using.

Failing to use a BUFG can force a clock onto local routing resources and the clock skew between different flip-flops can cause them to accept the wrong data or become metastable. Failing to use one of the Schmitt-triggered inputs for single-stepping can cause strange behavior because the slow, noisy clock edge on the other parallel port pins can cause multiple triggers on each edge. Either or both of these problems may afflict your circuit. Hard to say without seeing the VHDL and pin assignments.
